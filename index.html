What Orbit Chat is

A focused desktop client for direct and group messaging. Built with Electron + React for fast iteration, with encrypted messaging, encrypted media, mention/reaction UX, passcode controls, and social workflows.

Built for social chat

Orbit Chat keeps the experience direct: jump from home recent chats into DMs or groups, use dedicated tabs for each surface, react quickly, mention with autocomplete, and move older threads into archive.

Security-first defaults

Message APIs use ciphertext + nonce payloads. Conversation keys are sealed per participant, membership changes rotate key versions, and payload decryption remains client-side.

Desktop production UX

Scroll-safe scenes, passcode context labels, settings-driven chat naming, and hardened endpoint URL handling improve reliability in packaged desktop deployments.

Media and link support

Image attachments are encrypted client-side before upload. Pasted links stay inside the encrypted message body, and GIF sharing is available from search.

Conversation quality-of-life

Unread counters clear when a chat becomes active, while per-chat settings let users update display names and lock/passcode behavior in one place. Message pinning and in-chat search support quick context jumps.

Profile and social flow

Orbit includes profile popovers/settings, friend list refresh events, and account recovery code paths for login and passcode bypass.

How encryption works

Orbit Chat uses libsodium to encrypt direct and group messages end-to-end. Conversation keys are created on the client and encrypted to each participant’s public key, so the server stores opaque payloads.

End-to-end encrypted chat content
Direct and group messages are encrypted on your device before sending. The server receives and stores ciphertext + nonce plus an encrypted conversation key per user. Membership changes rotate group key versions, and history is constrained by each member’s readable key-version boundary.
1

Generate device keys

Each device generates a public/private keypair locally. The public key is registered with your account.

2

Seal a conversation key

For each conversation, the client creates a symmetric key and encrypts it to each participant’s public key (sealed box).

3

Encrypt messages, decrypt locally

Messages are encrypted client-side using the conversation key (libsodium secretbox). Group membership changes trigger key-version rotation, and recipients decrypt locally.

Encrypted End-to-End

  • Direct message text payloads
  • Group message text payloads
  • Attachment metadata in encrypted envelopes
  • GIF links embedded in encrypted envelopes
  • @mention text embedded in encrypted payloads
  • Wrapped file keys per attachment
  • Attachment blob bytes uploaded as encrypted chunks

Not Encrypted End-to-End

  • Conversation membership and who you talk to
  • Message timestamps and delivery/seen metadata
  • Profile and friendship metadata
  • Media reservation routing/storage metadata
  • Attachment lifecycle status metadata

Architecture and Runtime

Orbit Chat uses realtime room fanout, encrypted message envelopes, membership-change rekeying, and deterministic local state updates so delivery and unread behavior remain consistent.

Realtime delivery model

Messages are emitted to conversation rooms and user rooms (safety net). Client upsert is id-based to avoid duplicate timeline rows.

Chat identity model

Chats use custom display names when set, with fallback labels such as @username#chatId. Home view surfaces recent activity first.

Passcode and lock context

Passcode prompts include chat labels to reduce unlock mistakes across multiple DMs with the same person.

Conversation interaction layer

Message context actions support reply, ping, thread, react, and pin flows, with reaction updates fanned out in realtime.

First-message reliability

New inbound DM traffic appears in realtime without requiring a re-login refresh, including first-time recipient delivery paths.

Key-prep fallback behavior

If key material is still preparing on receive, UI may briefly show encrypted fallback text and then resolve to plaintext once unsealed; older-than-boundary versions stay hidden.

Endpoint normalization

Client URL normalization hardens malformed API/socket configuration values to reduce local setup and desktop deployment mistakes.

Realtime resilience

Socket transport starts with polling and upgrades to websocket, with silent token refresh support during auth-related reconnect failures.

Current Limits

Honest status of today’s model and roadmap-adjacent constraints.

Group E2EE currently depends on centralized membership enforcement and server-coordinated key distribution metadata.
Metadata remains visible to backend services.
Private keys are in local app storage, not OS keychain.
User fingerprint verification is not implemented yet.
Forward secrecy / ratcheting is not implemented yet.
Pinned chats and pinned messages are local-only and currently not synced across devices.
Large installer artifacts may require LFS/CDN strategy as release size grows.
In-app desktop auto-update is not enabled yet; installs still come from release artifacts.
